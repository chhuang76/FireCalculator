# Technical Design Document: Stateless Retirement Monte Carlo Simulator

---

## 1. System Overview

This project aims to build a **stateless, client-side web application** that allows users to simulate the success rate of various portfolio configurations against complex, multi-phase spending schedules over long time horizons (e.g., 55 years). 

The tool utilizes a **Monte Carlo simulation** engine running entirely in the user's browser, eliminating the need for a backend, database, or user accounts. 

### Core Guiding Principles
* **Zero Footprint:** No user data is stored on any server.
* **Vibe Coding Friendly:** The architecture is kept modular and simple so an AI agent can easily generate, debug, and iterate on the code.
* **High Fidelity:** Despite being a browser tool, the math handles asset correlations, multi-phase spending, and inflation variance accurately.

---

## 2. Architecture & Tech Stack

To maintain the "stateless" requirement while providing a modern, responsive UI, the following stack is proposed:

* **Framework:** Vite + React (lightweight and optimized for client-side applications).
* **Styling:** Tailwind CSS (for fast, utility-first layout building).
* **Math & Stats:** `mathjs` or `simple-statistics` for matrix operations (Cholesky decomposition).
* **Charts:** `Chart.js` or `Recharts` to render the Monte Carlo "spaghetti plot" of portfolio trajectories.
* **Browser Support:** Modern browsers only (Chrome, Firefox, Safari, Edge - last 2 versions).

## 3. Data Strategy: Fetching & Handling Stock Data

Because the app is stateless and runs in the browser, calling live financial APIs (like Alpha Vantage or Yahoo Finance) directly is problematic due to CORS restrictions and the exposure of private API keys.

### Proposed Hybrid Approach

1.  **Historical Data Fetching (Primary Method):**
    The app will fetch historical price data for user-selected assets to calculate:
    * **Expected Annual Return ($\mu$)**
    * **Annual Volatility ($\sigma$)**
    * **Pairwise Correlation Matrix ($\rho$)** - calculated from historical returns of all assets in the portfolio

    * **Algorithm to convert raw data to inputs:** Historical monthly prices will be used to calculate:
        * $Log\ Returns = \ln(P_t / P_{t-1})$
        * $\mu = \text{mean}(Log\ Returns) \times 12$
        * $\sigma = \text{std\_dev}(Log\ Returns) \times \sqrt{12}$
        * $\rho_{i,j} = \text{correlation}(Returns_i, Returns_j)$

2.  **The Local Defaults Table (Fallback):**
    The app will maintain a JSON dictionary of major asset classes and popular tickers (VT, QQQ, AVUV, BND, GLD, BTC) with pre-calculated values as fallback when historical data is unavailable.

3.  **User Overrides:**
    If a user enters a ticker not in the local dictionary and historical data cannot be fetched, the UI will prompt them to manually enter an estimated return and volatility.

## 4. The Simulation Algorithm (Under the Hood)

The simulation runs a loop for $N$ iterations (e.g., 5,000 or 10,000). For each iteration, it simulates the portfolio balance year-by-year across the total requested timeline.

### Step 1: Handling Asset Correlation
To simulate realistic market conditions where stocks crash together but gold or cash might stay flat, we cannot simply generate independent random returns for each asset. We must use a **Multivariate Normal Distribution**.

1.  **Define the Correlation Matrix ($R$):** An $n \times n$ matrix defining how the $n$ assets move together. The correlation matrix will be **calculated from historical price data** when available for the selected assets.
2.  **Define the Covariance Matrix ($\Sigma$):** Transformed from correlation using the standard deviations ($\sigma$) of each asset:
    $$\Sigma_{i,j} = R_{i,j} \times \sigma_i \times \sigma_j$$
3.  **Cholesky Decomposition:** Since drawing from a multivariate normal distribution directly in JS can be slow without heavy libraries, we use the Cholesky decomposition ($L$) of the covariance matrix, where $\Sigma = L L^T$.
4.  **Generating Correlated Returns:** Every simulated year, we generate a vector of independent standard normal random variables $Z \sim N(0,1)$. We then calculate the correlated returns vector $X$ as:
    $$X = \mu + L Z$$
    Where $\mu$ is the vector of expected returns for the assets.

### Step 2: The Multi-Phase Spending Engine
The spending engine maps the array of "Phases" into a flat array where index `i` represents the real dollar spend for year `i`.

**Logic Example:**
* Input: Phase 1 ($50k for 15 years), Phase 2 ($30k for 40 years).
* Flattened Array: `[50k, 50k ... (15 times) ... 30k, 30k ... (40 times)]`

During the simulation loop for Year $t$:
1.  **Inflation adjustment:** Determine cumulative inflation $I_t = (1 + \text{inflation\_rate})^t$, where inflation_rate is the fixed rate (default 3%) provided by the user.
2.  **Nominal Spend:** $S_{nominal} = \text{FlattenedSpend}[t] \times I_t$.

### Step 3: The Year-by-Year Loop
For each simulation (run $j$ from $1$ to $N$):
1.  Set $Portfolio\_Balance = \text{Initial Deposit}$.
2.  For each year $t$ from $1$ to $\text{Total Years}$:
    * Generate the vector of correlated returns $X$ for that year.
    * Calculate the total portfolio growth factor: $G = \sum (\text{Weight}_i \times X_i)$.
    * Apply growth: $Portfolio = Portfolio \times (1 + G)$.
    * Deduct Nominal Spending: $Portfolio = Portfolio - S_{nominal}$.
    * **Annual Rebalancing:** Reset asset allocations to their target weights based on the new portfolio balance.
    * If $Portfolio \le 0$ (including mid-year), mark this simulation run as a **Failure** and break the loop.
3.  If the portfolio balance remains positive through all years, mark the run as a **Success**.
4.  Calculate **Success Rate** = $(\text{Total Successful Runs} / N) \times 100$.

## 5. UI/UX Component Breakdown

### Component 1: Portfolio Setup
* **Table Rows:** Dynamic array of inputs.
* **State:** Array of objects `[{ ticker: 'VT', value: 200000, return: 0.09, vol: 0.17 }]`.
* **Input Format:** Users enter dollar values ($) for each asset, not percentages.
* **Auto-Calculations:** Displays running total of portfolio value and percentage weight of each row in real-time (calculated from $ values).
* **Validation:** Initial deposit must be greater than zero to run simulation.

### Component 2: Multi-Phase Spending Builder
* **Inputs:** `[ Spending Amount ]` and `[ Duration in Years ]`.
* **Action:** "Add Phase" button creates a new row.
* **Design:** Phases are entered in a continuous sequence. Each new phase automatically starts where the previous phase ended, preventing gaps or overlaps.
  * Example: Phase 1 (15 years) covers Years 1-15. Phase 2 (30 years) automatically starts at Year 16 and covers Years 16-45.
* **Validation:** Spending amounts must be positive (greater than zero).
* **Guardrails:** Total years automatically sums up at the bottom to show the user the total duration of their retirement.

### Component 3: Configuration & Execution
* **Inputs:**
  * Expected baseline inflation (user input, default: 3%, fixed rate for Phase 1)
  * Number of simulations (dropdown: 1k, 5k, 10k)
* **Action:** "Run Simulation" button.
* **State:** Triggers the Web Worker (optional but recommended for large $N$ so the UI doesn't freeze) to run the math loop.

### Component 4: Results Display (Output)
* **Hero Stat:** Large percentage success rate with a color gradient (Red < 70%, Yellow 70-90%, Green > 90%).
* **Statistics Panel:**
  * Median ending balance
  * Percentile bands (10th, 25th, 75th, 90th percentiles)
  * Worst-case scenario balance
  * Years until failure (for failed runs)
* **The Line Chart:** Displays percentile bands (10th, 25th, 50th/median, 75th, 90th) showing the distribution of portfolio values over time.

## 8. Decisions Made

### 8.1 Inflation Model
**Decision:** ✅ Phase 1 uses a fixed inflation rate with user input (default: 3%). Stochastic/historical inflation modeling will be added in Phase 2.

### 8.2 Correlation Matrix Calculation
**Decision:** ✅ Correlation matrix will be calculated from historical price data when available. This requires fetching historical price data for all assets in the portfolio and computing pairwise correlations.

### 8.3 Tech Stack
**Decision:** ✅ Vite + React (lightweight and optimized for pure client-side applications).

### 8.4 Edge Cases & Validation Rules
**Decision:** ✅ Validation rules established:
- **Negative spending amounts:** Show error message and prevent simulation from running.
- **Zero initial deposit:** Prevent simulation from running (validation: initial deposit must be > 0).
- **Spending phases with gaps or overlaps:** Not possible by design—UI enforces continuous sequence of phases.
- **Asset weights not summing to 100%:** Not applicable—users enter dollar values ($), not percentages. Weights are auto-calculated.
- **Portfolio goes negative mid-year:** Treated as a failed run; simulation stops for that iteration.

### 8.5 Portfolio Rebalancing
**Decision:** ✅ Phase 1 defaults to annual rebalancing. Each year, after growth and spending, the portfolio resets asset allocations to their target weights. Quarterly rebalancing and other options will be added in Phase 2.

### 8.6 Browser Compatibility
**Decision:** ✅ Support modern browsers only (Chrome, Firefox, Safari, Edge - last 2 versions).

### 8.7 Results & Statistics Display
**Decision:** ✅ Display comprehensive statistics beyond success rate:
- Median ending balance
- Percentile bands (10th, 25th, 75th, 90th percentiles)
- Worst-case scenario balance
- Years until failure (for failed runs)

### 8.8 Chart Visualization
**Decision:** ✅ Phase 1 will display percentile bands only (10th, 25th, 50th/median, 75th, 90th) on the chart to keep visualization clean and simple. Individual simulation paths may be added in Phase 2.
