# Technical Design Document: Stateless Retirement Monte Carlo Simulator

**Version:** 1.0
**Status:** ✅ Complete - Ready for Implementation
**Last Updated:** 2026-03-31

---

## Executive Summary

A **stateless, client-side web application** for simulating retirement portfolio success rates using Monte Carlo analysis. The tool runs entirely in the browser with no backend, database, or user accounts.

**Key Features:**
- Multi-phase spending schedules (e.g., different spending in early vs late retirement)
- Multi-asset portfolios with realistic correlation modeling
- 5,000-10,000 Monte Carlo simulations per run
- Visual results with percentile bands and comprehensive statistics
- 100% offline capability using pre-fetched historical data

**Tech Stack:** Vite + React + Tailwind CSS + Web Workers
**Target Users:** Individual investors planning retirement
**Cost:** $0 (fully open source, no runtime costs)

## 3. Data Strategy

### Static Historical Data Files

**Approach:** Pre-fetch historical price data and store as static CSV files in the repository.

#### Why Static Files?

Traditional approach (live API calls) has major drawbacks:
- ❌ CORS restrictions block browser calls
- ❌ API keys exposed in client code (security risk)
- ❌ Rate limits (most free tiers: 25-800 calls/day)
- ❌ Network latency on every load
- ❌ Requires backend proxy (adds complexity)

**Static files solve all of these:**
- ✅ No API calls at runtime (instant loading)
- ✅ No CORS issues
- ✅ No rate limits
- ✅ Works completely offline
- ✅ Truly stateless (no backend)
- ✅ Zero runtime costs

#### Data File Structure

```
public/data/
├── VT.csv      # Vanguard Total World Stock
├── QQQ.csv     # Nasdaq 100
├── AVUV.csv    # Avantis US Small Cap Value
├── BND.csv     # Vanguard Total Bond
├── GLD.csv     # Gold ETF
└── BTC.csv     # Bitcoin
```

**CSV Format (Phase 1):**
```csv

2015-02-01,46.01
2015-03-01,46.89
```

**File Size:** ~300KB total for all 6 assets (negligible)

#### Supported Assets (Phase 1)

| Ticker | Name | Inception | Est. Data Points |
|--------|------|-----------|------------------|
| VT | Vanguard Total World | 2008 | ~200 months |
| QQQ | Nasdaq 100 ETF | 1999 | ~300 months |

#### Data Update Process

**Script:** `scripts/update-data.js`

**Process:**
1. Developer manually runs script monthly (e.g., first day of month)
2. Script fetches latest data from Twelve Data API (or similar)
3. Converts to CSV format (`date,close`)
4. Overwrites existing files in `public/data/`
5. Developer commits updated CSVs to git

**Frequency:** Monthly (historical monthly data rarely needs updates)

**Automation:** Not needed for Phase 1 (manual is sufficient)

**Script Design Requirements:**
- Configurable field mapping (easy to add `open,high,low,volume` later)
- Error handling for API failures
- Validation of data integrity
- Backwards compatible (old CSV format still works)

#### Runtime Calculation

When user adds an asset to portfolio:

1. **Load CSV:** Fetch `/data/VT.csv` via standard HTTP request
2. **Parse:** Convert CSV to array of `{date, price}` objects
3. **Calculate Log Returns:** $\text{logReturn}_i = \ln(P_i / P_{i-1})$
4. **Annualized Return:** $\mu = \text{mean}(\text{logReturns}) \times 12$
5. **Annualized Volatility:** $\sigma = \text{std}(\text{logReturns}) \times \sqrt{12}$
6. **Correlation Matrix:** After all assets loaded, calculate pairwise correlations: $\rho_{ij} = \text{corr}(\text{returns}_i, \text{returns}_j)$

**Caching:** Browser automatically caches CSV files (304 Not Modified on reload)

#### Limitations & Future Work

**Phase 1 Limitations:**
- Only 6 pre-defined tickers
- Data becomes stale (requires monthly updates)
- No support for custom/arbitrary tickers

**Phase 2 Enhancements:**
- Optional serverless proxy for custom tickers
- User can input custom return/volatility if ticker not found
- Automated data refresh via GitHub Actions

## 4. The Simulation Algorithm

### Overview

For each Monte Carlo run, simulate the portfolio year-by-year from retirement start to end. Track whether the portfolio survives (success) or depletes (failure).

**Key Insight:** Must use correlated returns, not independent random draws. In real markets, VT and QQQ crash together; BND and GLD stay stable.

### Step 1: Correlation & Covariance Matrix

#### Define Correlation Matrix ($R$)

An $n \times n$ matrix where $R_{ij}$ represents correlation between assets $i$ and $j$.

**Example (3 assets):**
```
       VT    QQQ   BND
VT   [1.00  0.95  0.10]
QQQ  [0.95  1.00  0.15]
BND  [0.10  0.15  1.00]
```

Interpretation:
- VT and QQQ highly correlated (0.95) — both are stocks
- VT and BND weakly correlated (0.10) — stock/bond diversification

#### Calculate Covariance Matrix ($\Sigma$)

Convert correlation to covariance using asset volatilities:

$$\Sigma_{ij} = R_{ij} \times \sigma_i \times \sigma_j$$

#### Cholesky Decomposition

Decompose $\Sigma = L L^T$ where $L$ is lower triangular.

**Why?** Allows efficient generation of correlated random returns.

**Library:** Use `mathjs` or `simple-statistics` for Cholesky decomposition in JavaScript.

#### Generate Correlated Returns

For each simulated year:

1. Generate independent standard normal random variables: $Z \sim N(0, 1)$ (one per unique asset)
2. Transform to correlated returns: $X = \mu + L \cdot Z$

Where:
- $\mu$ = vector of expected returns $[\mu_1, \mu_2, ..., \mu_n]$
- $L$ = Cholesky decomposition of covariance matrix
- $Z$ = vector of independent standard normals
- $X$ = resulting correlated returns for this year

**Result:** $X$ contains realistic returns where assets move together according to historical correlations.

**Pre-Processing: Aggregate Duplicate Tickers**

Before simulation starts, aggregate portfolio entries by ticker:

```javascript
// User input
const input = [
  { ticker: 'VT', value: 100000 },
  { ticker: 'VT', value: 50000 },
  { ticker: 'BND', value: 75000 }
];

// Aggregated (used for simulation)
const aggregated = [
  { ticker: 'VT', value: 150000 },  // Combined
  { ticker: 'BND', value: 75000 }
];
```

This is mathematically equivalent but more efficient:
- Fewer positions to track during simulation
- Correlation matrix is smaller (2×2 instead of 3×3)
- Same result: `VT₁ × (1+r) + VT₂ × (1+r) = (VT₁ + VT₂) × (1+r)`

### Step 2: Multi-Phase Spending Engine

Users define spending in phases, e.g.:
- Phase 1: $50,000/year for 15 years (ages 55-70, active retirement)
- Phase 2: $30,000/year for 40 years (ages 70-110, reduced spending)

**Implementation:** Flatten into array indexed by year:

```javascript
// Input: [{amount: 50000, years: 15}, {amount: 30000, years: 40}]
// Output: [50k, 50k, ...(15x)..., 30k, 30k, ...(40x)...]
const flatSpending = phases.flatMap(p => Array(p.years).fill(p.amount));
```

**Inflation Adjustment:**

For year $t$:

$$I_t = (1 + r_{inflation})^t$$

$$S_{nominal}(t) = S_{real} \times I_t$$

Where:
- $r_{inflation}$ = fixed annual inflation rate (default 3%)
- $S_{real}$ = real dollar spending amount (from phase definition)
- $S_{nominal}(t)$ = inflation-adjusted spending in year $t$

**Example:**
- Phase 1: $50,000 real dollars
- Year 1: $50,000 \times 1.03^1 = $51,500$
- Year 10: $50,000 \times 1.03^{10} = $67,196$

### Step 3: Year-by-Year Simulation Loop

**Outer Loop:** Run $N$ simulations (e.g., 10,000)

**Inner Loop:** For each simulation, iterate through years 1 to $T_{total}$

#### Pseudocode

```
# Pre-processing (before simulation loop)
Aggregated_Portfolio = AggregateDuplicateTickers(User_Input)
# Example: [VT:$100k, VT:$50k, BND:$75k] → [VT:$150k, BND:$75k]

Initial_Deposit = Sum(Aggregated_Portfolio values)
n = Number of unique tickers

For each simulation run j = 1 to N:
    Portfolio_Balance = Initial_Deposit

    For each year t = 1 to Total_Years:
        # 1. Generate returns for this year (one per unique ticker)
        Z = vector of n independent standard normals
        X = μ + L · Z    # Correlated returns

        # 2. Calculate portfolio growth
        G = Σ(Weight_i × X_i)    # Weighted average return
        Portfolio = Portfolio × (1 + G)

        # 3. Deduct spending
        Inflation_Factor = (1 + inflation_rate)^t
        Spending = FlatSpending[t] × Inflation_Factor
        Portfolio = Portfolio - Spending

        # 4. Rebalance to target weights
        # (Reset dollar amounts to maintain target % allocations)
        RebalancePortfolio()

        # 5. Check for failure
        If Portfolio ≤ 0:
            Mark run as FAILURE
            Record failure_year = t
            Break inner loop

    If Portfolio > 0 at end:
        Mark run as SUCCESS
        Record ending_balance

# Calculate statistics
Success_Rate = (Successful_Runs / N) × 100%
```

#### Annual Rebalancing

After growth and spending each year, reset asset allocations to target weights.

**Example:**
- Target: 60% VT ($120k), 40% BND ($80k)
- After Year 1: VT grew to $135k, BND to $75k → Total = $210k
- Rebalance: Sell $9k VT, buy $9k BND → VT = $126k (60%), BND = $84k (40%)

**Implementation:**
```javascript
function rebalance(portfolio, targetWeights) {
  const total = portfolio.reduce((sum, asset) => sum + asset.value, 0);
  portfolio.forEach(asset => {
    asset.value = total * targetWeights[asset.ticker];
  });
}
```

**Phase 2:** Add options for no rebalancing, quarterly rebalancing, etc.

### Step 4: Track Statistics Across Simulations

For each simulation run, store:
- Success/failure status
- Ending balance (if success)
- Year of failure (if failure)
- Portfolio balance at each year (for percentile calculations)

**Storage Structure:**
```javascript
const results = {
  runs: [
    { success: true, endingBalance: 1250000, balancesByYear: [...] },
    { success: false, failureYear: 23, balancesByYear: [...] },
    // ... 10,000 entries
  ]
};
```

**Post-Processing:**

Calculate aggregate statistics:
- **Success Rate:** % of runs where portfolio survived
- **Median Ending Balance:** 50th percentile of successful runs
- **Percentile Bands:** 10th, 25th, 50th, 75th, 90th percentiles at each year
- **Worst Case:** Minimum ending balance among successes
- **Years to Failure:** Average/median for failed runs

## 5. UI/UX Component Breakdown

### Component 1: Portfolio Setup

**Purpose:** Define initial portfolio composition

**UI Elements:**
- Dynamic table with "Add Asset" button
- Each row: [Ticker Dropdown] [$Amount Input] [Remove Button]
- Ticker dropdown: VT, QQQ, AVUV, BND, GLD, BTC
- Auto-calculated fields (read-only):
  - Total portfolio value
  - % weight of each row (based on individual values)
  - Expected return (calculated from CSV data)
  - Volatility (calculated from CSV data)

**Display Notes:**
- Table shows individual entries exactly as user entered them (including duplicates)
- If same ticker appears multiple times, each row displays separately
- User sees: "VT: $100,000 (40%)" and "VT: $50,000 (20%)" as separate rows
- Behind the scenes: aggregated to VT: $150,000 (60%) for simulation

**State Management:**
```javascript
// User input (allows duplicates for easy data entry)
const [portfolioInput, setPortfolioInput] = useState([
  { ticker: 'VT', value: 200000, mu: 0.09, sigma: 0.17 },
  { ticker: 'BND', value: 100000, mu: 0.04, sigma: 0.05 },
  { ticker: 'VT', value: 50000, mu: 0.09, sigma: 0.17 }  // Duplicate OK
]);

// Aggregated for simulation (duplicates combined)
const aggregatedPortfolio = [
  { ticker: 'VT', value: 250000, mu: 0.09, sigma: 0.17 },  // $200k + $50k
  { ticker: 'BND', value: 100000, mu: 0.04, sigma: 0.05 }
];
```

**Note:** Duplicates are allowed for data entry (users can copy-paste from multiple accounts). Before running simulation, duplicate tickers are aggregated by summing their values. This is mathematically equivalent and computationally more efficient.

**Validation:**
- Total portfolio value must be > $0
- Each asset value must be > $0
- Duplicate tickers are allowed (users may have same ticker in multiple accounts)
- Show error messages inline

**Data Loading:**
When user selects ticker from dropdown:
1. Fetch `/data/{TICKER}.csv` (or use cached data if already loaded)
2. Calculate μ and σ from historical returns
3. Update portfolio state with calculated values
4. Display "Loading..." state during fetch

**Handling Duplicates:**
If user adds the same ticker multiple times (e.g., VT in 401k and IRA):
- CSV is only fetched once (cached after first load)
- Display shows individual entries (as user entered them)
- Before simulation, aggregate by ticker: sum all values for same ticker
- Example: `[VT: $100k, VT: $50k, BND: $75k]` → `[VT: $150k, BND: $75k]`
- Simulation runs on aggregated portfolio (more efficient)

### Component 3: Configuration & Execution

**Purpose:** Set simulation parameters and run the simulation

**UI Elements:**
- **Inflation Rate:** Input field (default: 3%) with % label
- **Number of Simulations:** Dropdown (1,000 / 5,000 / 10,000)
- **Run Simulation:** Large primary button

**Pre-Flight Validation:**

Before allowing simulation to run, check:
- ✅ Portfolio has at least 1 asset
- ✅ Total portfolio value > $0
- ✅ At least 1 spending phase defined
- ✅ All spending amounts > $0
- ✅ Inflation rate is a valid number

If validation fails, disable button and show error messages.

**Execution Flow:**
1. User clicks "Run Simulation"
2. Pre-process portfolio: aggregate duplicate tickers
   ```javascript
   function aggregatePortfolio(portfolio) {
     const aggregated = {};
     portfolio.forEach(asset => {
       if (aggregated[asset.ticker]) {
         aggregated[asset.ticker].value += asset.value;
       } else {
         aggregated[asset.ticker] = { ...asset };
       }
     });
     return Object.values(aggregated);
   }
   ```
3. Button shows loading state: "Simulating... X%"
4. Spawn Web Worker with aggregated portfolio
5. Worker sends progress updates every 10%
6. Worker returns results when complete
7. Display results in Component 4

**Web Worker Communication:**
```javascript
// Main thread
const worker = new Worker('simulation-worker.js');

// Aggregate portfolio before sending to worker
const aggregated = aggregatePortfolio(portfolioInput);

worker.postMessage({
  portfolio: aggregated,  // Aggregated portfolio (duplicates combined)
  phases: [...],
  inflation: 0.03,
  iterations: 10000
});

worker.onmessage = (e) => {
  if (e.data.type === 'progress') {
    setProgress(e.data.percent);
  } else if (e.data.type === 'complete') {
    setResults(e.data.results);
  }
};
```

### Calculation Examples

**Annualized Return from Monthly Data:**
```
Monthly returns: [0.02, -0.01, 0.03, 0.01, ...]
Mean monthly return: 0.0075
Annualized return: 0.0075 × 12 = 0.09 (9%)
```

**Annualized Volatility from Monthly Data:**
```
Std dev of monthly returns: 0.05
Annualized volatility: 0.05 × √12 = 0.173 (17.3%)
```

**Correlation Calculation:**
```
VT monthly returns: [0.02, -0.01, 0.03, ...]
QQQ monthly returns: [0.025, -0.015, 0.032, ...]
Correlation(VT, QQQ) = 0.95 (highly correlated)
```

**Portfolio Aggregation (Duplicate Tickers):**
```
User Input:
[VT: $100,000, VT: $50,000, BND: $75,000]

Aggregated for Simulation:
[VT: $150,000, BND: $75,000]

Mathematical Equivalence (Year 1 with VT return = +10%, BND return = +3%):

Method 1 (separate):
VT₁: $100k × 1.10 = $110k
VT₂: $50k × 1.10 = $55k
BND: $75k × 1.03 = $77.25k
Total: $242.25k

Method 2 (aggregated):
VT: $150k × 1.10 = $165k
BND: $75k × 1.03 = $77.25k
Total: $242.25k

✓ Same result, but Method 2 is more efficient
```
